# Product Fixture Gate Resolution

Date: 2026-05-15

## Question

Should the top-ranked product fixture gates be resolved by accepting disclosed generated fixtures, or by adding public binary replacements?

## Hypothesis

For the current guarded release-proof wording, product can accept disclosed generated fixtures for package-topology edge cases. Public binary replacements should remain required before stronger real-world, vendor-specific, trust, malware, signed-provenance, or Excel-repair wording.

## External sources checked

- Microsoft Protected View describes unsafe files as a review/trust decision, not a package-topology claim: https://support.microsoft.com/en-us/office/what-is-protected-view-d6f09ac7-e6b9-4495-8e43-2bbcdbcb6653
- Microsoft Open Packaging Conventions describe package parts, relationships, and digital signatures: https://learn.microsoft.com/en-us/previous-versions/windows/desktop/opc/open-packaging-conventions-overview
- Apache POI documents OOXML signing as generated package/signature behavior that requires extra crypto dependencies: https://poi.apache.org/encryption.html
- openpyxl documents preservation limits for unsupported workbook objects: https://openpyxl.readthedocs.io/en/stable/tutorial.html

## Why this matters to Ascend

The release proof index now ranks fixture policy as the first owner action for both top claims. If the owner loop treats every generated edge package as unacceptable, the proof program stalls waiting for rare public signed/unknown fixtures. If it accepts generated fixtures without disclosure, the release claim overstates evidence quality.

## Probe/implementation

I ran the checked-in public fixture scan:

```bash
bun run fixtures/benchmarks/safe-open-fixture-scan.ts --json
```

Result:

- scanned: 351 public XLSX/XLSM fixtures
- rejected during scan: 2
- signed/unknown replacement candidates: 0
- replacement status: `no-public-binary-replacement-found`

I also inspected `defaultPackageActionProofCases()`:

- public fixtures: 2 cases (`macro-passthrough`, `chart-sidecar-accounting`)
- generated workbooks: 2 cases (`regenerate-existing-sheet`, `add-sheet-part`)
- generated edge packages: 4 cases (`docprops-passthrough`, `calc-chain-drop`, `signature-invalidation-drop`, `unknown-part-error`)

No production code was changed.

## Results

The evidence supports a two-tier policy:

1. Accept disclosed generated fixtures for current guarded local proof when the edge case is structural package topology and generated by tracked harness code.
2. Require public binary fixtures before stronger wording that depends on real-world authoring behavior, vendor semantics, user trust, malware scanning, signed provenance, or Excel repair behavior.

The current safe-open and package-action claims remain blocked until product explicitly accepts that policy, but the next owner action should be policy approval rather than an unbounded public fixture hunt.

## Confidence

Medium-high. The local scan is bounded to checked-in public fixtures, and the external references support the distinction between package-topology proof and trust/security/provenance claims.

## Fold-in decision

Folded into the release claim board as a fixture gate resolution recommendation. No new surface or production behavior changed.
