# Public Latency Generated Edit Input

## Question

Can the public-tracked practical latency preset run edit/verify steps without depending on private workbook inputs, while still labeling generated inputs honestly?

## Hypothesis

Yes. A generated edit workbook produced by tracked benchmark code is better release-proof hygiene than silently skipping edit steps or falling back to private corpus files. It must remain labeled as generated evidence, not public real-workbook evidence.

## External sources checked

- hyperfine documents warmup runs and JSON/Markdown export, reinforcing that benchmark artifacts should report setup and run assumptions rather than just numbers: https://github.com/sharkdp/hyperfine
- Bun documents benchmarking/profiling workflows, including CPU and heap profiling options that support repeatable local benchmark evidence: https://bun.sh/docs/project/benchmarking
- SLSA provenance distinguishes generated byproducts and build inputs from signed provenance; generated local benchmark inputs should not be described as attested release evidence: https://slsa.dev/spec/v1.0-rc1/provenance

## Why this matters to Ascend

The release claim board keeps practical latency reports excluded from release proof until a tracked-clean public-input run exists with owner-approved wording. The public-tracked preset previously avoided private read inputs, but edit/verify evidence still needed a non-private workbook. A tracked-harness generated edit workbook lets performance owners run the whole practical latency contract locally without private data, while preserving the honest boundary.

## Probe/implementation

Finished the in-flight `fixtures/benchmarks/practical-latency-contracts.ts` change:

- public-tracked preset now resolves a generated edit workbook when `--edit-input-file` is absent;
- generated workbook bytes come from `buildRawReadWorkloadDataSet('mixed-10pct-text', 65536, 10)`;
- workflow/post-write/phase-profile commands receive the generated `--input-file`;
- input provenance records the edit input as `kind: generated` and `releaseClaimable: true`;
- Markdown labels generated inputs separately from tracked public fixtures;
- the focused test asserts generated edit input command plumbing and provenance.

Command run:

```bash
bun test fixtures/benchmarks/practical-latency-contracts.test.ts
```

## Results

- Focused test passed: 1 test, 13 assertions.
- The public-tracked dry run now includes a generated edit workbook path in edit/verify commands.
- First-view and table-inspection inputs remain tracked public fixtures.
- The edit input is explicitly `generated`, so performance reports can remain claim-safe without pretending the generated workbook is a real public fixture.

## Confidence

Medium-high. The dry-run path and command/provenance plumbing are covered. A timed full public-tracked run should still be owner-controlled because benchmark numbers depend on local hardware and current worktree state.

## Fold-in decision

Fold into the performance benchmark harness. Keep practical latency evidence excluded from release proof until a tracked-clean public-input run and threshold wording are approved.
